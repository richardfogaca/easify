Agor queues tasks rather than individual messages. When a session is already executing a task, later prompts and completion callbacks can be stored as queued tasks. Creation means the task exists durably; it does not mean execution has started. Queue processing later admits runnable work when the session is promptable. Documentation should explain those states without relying on incidental service names or promising that every queued task will drain immediately, because restarts and trigger conditions can affect the path.
